# Lesson 5: Pinning

## The problem: self-referential structs

When the compiler generates a state machine from `async fn`, it may create a struct that references its own fields:

```rust
async fn example() {
    let data = vec![1, 2, 3];
    let reference = &data;      // points to `data` above
    some_future().await;         // state machine must store both `data` and `reference`
    println!("{:?}", reference); // `reference` must still be valid
}
```

The generated state machine struct holds both `data` AND `reference`. If the struct is moved in memory, `reference` becomes a dangling pointer.

## Pin prevents moving

`Pin<&mut T>` is a wrapper that guarantees the value won't be moved:

- `Pin::new(&mut val)` — only works if `T: Unpin` (safe to move)
- `unsafe Pin::new_unchecked(&mut val)` — for non-Unpin types, you promise not to move it
- Once pinned, you can't get `&mut T` back (which would let you `std::mem::swap`)

Most futures are `!Unpin` (not safe to move) because they might be self-referential.

## Unpin

`Unpin` is an auto-trait that means "this type is safe to move even when pinned." Most regular types are `Unpin`. Futures generated by `async fn` are `!Unpin`.

```rust
// This is Unpin — no self-references, safe to move
struct Counter { count: u32 }

// This is !Unpin — compiler-generated, might be self-referential
async fn foo() { ... }
```

## The pin! macro

```rust
let future = async { ... };
pin!(future); // pins `future` to the stack, gives you Pin<&mut F>
future.poll(cx); // now you can poll it
```

## Exercises

### Exercise 1: Demonstrate the problem
Create a struct with a field and a raw pointer to that field (simulating self-reference). Move the struct. Show the pointer is now dangling (prints garbage or different address).

### Exercise 2: Pin it
Use `Pin` to prevent the move. Show that the compiler rejects code that tries to move a pinned value.

### Exercise 3: Implement a self-referential future
Create a future that holds a `String` and a reference to that `String` (using raw pointer or `PhantomPinned`). Implement `Future` for it. Show it works correctly when pinned.
